Why this check exists

Schema-conformant is not the same as true.

Protocols can standardize the shape of agent-readable data without checking whether its content matches the merchant’s records. The repo commits a feed that passes every pinned schema but is false against those records — the conformance leg passes it, the truth leg catches it. That pair is the product in one sentence.

How the check works

From the merchant’s record to a verdict with evidence.

  1. 1

    Ground truth

    The merchant’s own record

    The system of record holds the catalog, prices, and availability the restaurant itself maintains. Every claim in the serving copy is checked against it.

  2. 2

    What agents read

    The serving copies

    Platforms serve agent-readable copies of that record (ACP- and UCP-shaped feeds). This is what a shopping agent actually reads — and where the data can drift.

  3. 3

    Truth check

    Line-by-line comparison

    A deterministic comparator checks each claim in the copy against the record. It makes no AI calls, costs $0 to run, and produces the same report from the same input.

  4. 4

    Evidence

    Evidence for every finding

    Each finding includes the claim id, the rule that caught it, and the record row it was checked against — auditable end to end.

  5. 5

    Human gate

    Humans stay in the loop

    Findings become recommendations — reports, drafts, deliveries — behind explicit human gates. Nothing sends itself.

Three verification modules

Listings truth · protocol conformance · fee-cap audit.

  • Module

    Listings truth check

    The serving copy vs the system-of-record, line by line — availability, price, existence, encoding. The committed demo report catches a sold-out item still being served.

  • Module

    Protocol conformance

    Checks a feed against 78 pinned official UCP schemas (spec 2026-04-08). The committed feed passes every schema and is still false against the merchant’s records — the conformance leg passes it, the truth leg catches it. Both legs are machine-checked.

  • Module

    Fee-cap audit

    Codifies NYC §20-563.3 as 17 rules: 11 executable predicates and 6 registered non-checkable clauses. A calibrated fee-line classifier handles deliberately difficult, relabeled statement lines; the deterministic rule table makes the decision.

Labels must be earned

Every quality label traces to a pre-registered run.

Evaluation floors are committed in git before each run. Each split is closed after one scoring pass, and misses stay published. The evidence dashboard traces every published figure to its committed record.

  • Fee-line classifier

    21/21

    The classifier’s first run missed a required floor, so the label was deferred and the record stayed public. A fresh, pre-registered retry scored 21/21 and earned the label.

  • Agent-crew live run

    20/20

    The live crew run scored 20/20 against floors committed before the run. Only the two model-directed members carry the “agent” label, matching the pre-registration.

  • The lanes that missed

    Floors not met — and it says so

    The retrieval and entity-resolution lanes missed their pre-registered floors. Both labels remain deferred on every payload, and both scoreboards stay published.

What this is — and is not

A simulated prototype with clear limits.

The restaurant, menu, and records are invented. The verification rules, the pinned data-format standard, and the codified fee caps are real. This site is a static replay of committed, labeled fixtures and initiates nothing. The project began as a merchant-activation prototype; that first product is preserved, working, as the legacy activation module.
