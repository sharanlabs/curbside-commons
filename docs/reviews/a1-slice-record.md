# A1 slice record — MCP server (agentic extension, plan §5 row A1)

**Slice:** A1 (plan `docs/plan-agentic-extension.md` v1.0 §3–§6, §5 row A1). **Mode:** SPEC-ADHERENCE (delegated implementer lane; Fable orchestrates/judges — top-model-final). **Date:** 2026-07-07. **Status:** built + self-verified; **NOT committed** (awaits Fable equivalence review of the diff, per the standing 2026-07-03 bar → any downstream Codex batch → acceptance-gate). **Consumes:** A0 (`lib/tools/registry.ts`, committed at `2ae6654`) — read-only, never edited.

▸ *Plain: this slice builds a small "plug adapter" so any standard AI-tool client (Claude Desktop, an agent framework, an eval harness) can press the same six clearly-labeled buttons the A0 tool registry already built — over a private local pipe, never a network socket — using the exact rulebook the checker itself validates input against, not a hand-copied one. A real scripted conversation with the plug-in (list the buttons, press each one, press two wrong on purpose) is frozen byte-for-byte so anyone can prove it still behaves exactly as documented.*

## 1 · What was built

**Source (`lib/mcp/`):**
- `descriptions.ts` (57 lines) — the six honesty-labeled MCP tool descriptions (`TOOL_DESCRIPTIONS`), kept in their own module so the exact committed wording is one import away for the conformance tests. Every description states the underlying data is SIMULATED; `get_rule`'s description precisely scopes the one honest exception (its rule TABLE is the real published law, not simulated — only the audited statements/feeds elsewhere are); `run_demo`'s begins with `"DEMO-ONLY WALKTHROUGH — never an audit result"` verbatim; `classify_and_audit`'s states `"ADVISORY — candidate leads, never a verdict; classifier has NOT earned a calibrated label"` verbatim.
- `server.ts` (205 lines) — `createMcpServer()`: builds an unconnected MCP SDK `Server` (low-level API — see Escalation E-1 for why, not the high-level `McpServer.registerTool()`), registers `ListToolsRequestSchema` and `CallToolRequestSchema` handlers. `tools/list` advertises the six A0 tools, in the registry's own definition order, each with its committed input schema passed through VERBATIM (no hand re-authoring) and its committed description. `tools/call` runs `callTool(name, args)` — the registry's ONE execution path — and maps the result: `content: [{type:"text", text: result.canonical}]`, `structuredContent: {tool, ok, exitCode, demoOnly, advisory, earnsLabel}` (the chosen machine-readable slot; see Escalation E-2), `isError: false`. `exitCode !== 0` is never an MCP error. Error mapping: `ToolInputError` → `isError:true` with a message carrying every ajv error's JSON pointer; `ToolNotFoundError` → mapped the same way (defensive — unreachable through this server's own advertised tool list, but a raw `tools/call` with an arbitrary name is still possible over the wire); any other engine error (`StatementParseError`, an unreadable file, `RuleNotFoundError`) → `isError:true` with the engine's own message. Nothing is ever silently swallowed into a success.

**Launcher (`bin/mcp-server.mjs`, 26 lines):** mirrors `bin/check.mjs`'s dynamic-`import()`-of-a-`.ts`-file pattern (Node's native TS type-stripping) — builds the server from `lib/mcp/server.ts` and connects it to a `StdioServerTransport`. Zero network: stdio only, never an HTTP/SSE transport.

**Tests (`evals/mcp/`):**
- `harness.ts` (33 lines) — shared spawn helper: connects a fresh MCP SDK `Client` to `bin/mcp-server.mjs` as a real child process, one spawn per test FILE (`beforeAll`/`afterAll`), matching `cli-c1.test.ts`'s generous-timeout discipline (60s).

## 2 · Escalations (judgment calls for the orchestrator to overrule)

- **E-1 — low-level `Server` API used instead of the high-level `McpServer.registerTool()` convenience API.** The packet requires advertising "the committed A0 input schema verbatim" and explicitly says: "if the SDK API you use requires zod shapes, use its raw-JSON-schema registration path instead." Checked the installed SDK's own type declarations (`node_modules/@modelcontextprotocol/sdk/dist/esm/server/mcp.d.ts`): `McpServer.registerTool()`'s `inputSchema` config accepts only a Zod raw shape (`ZodRawShapeCompat`) or a Standard-Schema object (`AnySchema`) — never a plain JSON-Schema object — so using it would force hand-converting the committed schema into a different, re-authored shape. The low-level `Server` class's `setRequestHandler(ListToolsRequestSchema, ...)`/`setRequestHandler(CallToolRequestSchema, ...)` path takes a tool's `inputSchema` as the SDK's own `Tool` wire type, which IS a plain JSON-Schema-shaped object (`{type:"object", properties?, required?} & catchall unknown` — verified by reading `types.d.ts`'s `ToolSchema` directly, not assumed). Used the low-level path per the packet's own fallback instruction. **Overrule path:** none expected — this is the literal resolution the packet itself names for exactly this SDK-version condition.
- **E-2 — `structuredContent` chosen over `_meta` as the machine-readable slot.** The packet says "pick the SDK-sanctioned slot for machine-readable metadata and document the choice." `structuredContent` is the SDK's own `CallToolResultSchema`-defined slot (`Record<string, unknown>`) specifically for a tool's machine output; `_meta` is the protocol's own out-of-band envelope (progress tokens, related-task ids) per `types.d.ts`, not a tool-domain slot. Chose `structuredContent`, always with a FIXED shape (`tool`, `ok`, `exitCode`, `demoOnly`, `advisory`, `earnsLabel` — `earnsLabel: null`, not omitted, on the four tools with no advisory classifier at all) so a client never has to branch on which keys exist. **Overrule path:** a one-function edit (`structuredContentFor`) if `_meta` is preferred instead.
- **E-3 — the import-walk requirement ("every reachable repo file under lib/ is either lib/tools/** or lib/mcp/**") is literally unsatisfiable as a single TRANSITIVE walk, and was split into two tests.** `lib/tools/registry.ts` (A0, already committed, unchangeable by this slice) itself transitively imports `lib/packs/listings/cli.ts` and `lib/packs/fees/*` — that is A0's own design (the registry wraps unchanged engine entry points) and A0's own AC-3 proof (`registry-ac3-import-graph.test.ts`) already asserts this is fine (it only bans specific LLM/network patterns, never bans `lib/packs/**` itself). A literal full-transitive-closure reading of "every reachable lib/ file is lib/tools/** or lib/mcp/**" would therefore ALWAYS fail once the walk passes through the registry, regardless of anything this slice does. Read the sentence's load-bearing word as "directly" (per the SAME paragraph's own "NEVER lib/packs/** or lib/verifier-core/** directly"), and implemented it as two separate, honest tests: (1) a DIRECT-import-boundary check over files physically under `lib/mcp/**` only (no `lib/packs/**`/`lib/verifier-core/**` import anywhere in THIS slice's own source, and every repo-relative import target IS `lib/tools/**` or `lib/mcp/**`); (2) the TRANSITIVE $0/offline + zero-HTTP-transport proof, reusing the A0 walker verbatim, which legitimately reaches `lib/packs/**` — same as A0's own proof already does. **Overrule path:** if a stricter reading was actually intended (e.g., an entirely separate registry re-implementation with no packs reachability at all), that would be an architectural change to A0 itself, out of this slice's scope — flagged for the orchestrator, not resolved unilaterally.
- **E-4 — the MCP SDK reorders `Tool.inputSchema`'s keys on the wire; a canonicalizing (recursive key-sort) comparison was substituted for a raw `JSON.stringify` byte comparison.** Empirically verified (a smoke-test spawn + real client call, logged in this session before any test was written): a `tools/list` round-trip through the SDK client's own `ListToolsResultSchema.parse(...)` reconstructs each `inputSchema` with its Zod-declared keys (`type`, `properties`, `required`) first, then the JSON-Schema "catchall" keys (`$schema`, `$id`, `title`, `additionalProperties`) appended afterwards — a different key ORDER than the committed schema file, even though no key or value differs. A literal `JSON.stringify(a) === JSON.stringify(b)` would therefore spuriously fail on cosmetic ordering alone. Per the packet's own "if the SDK injects nondeterminism, normalize it ... and document exactly what is normalized" instruction, wrote `canonical-json.mjs` (recursive alphabetical key-sort; arrays never reordered, since `required`/`enum` are order-sensitive JSON Schema) and used it for BOTH the conformance test's schema comparison and the transcript writer's `tools/list` step. This is content-equality, not order-equality — arguably the more honest reading of "byte-equal" once one leg of the comparison has passed through a library that reorders keys it does not own. **Overrule path:** none expected — a raw byte comparison would fail on every single run regardless of any real drift, which is not useful signal; if a different (non-alphabetical) canonical order is preferred, `canonicalize()` is a one-function edit.
