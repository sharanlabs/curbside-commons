# T-001.5 Standards Blindspot Review

Review date: 2026-06-02
Reviewer: Claude Code (pre-flight blindspot review — review only; playbook not created)
Subject: the planned T-001.5 "Enterprise Professional Delivery Playbook" (15 planned additions).

Each recommendation is classified: **Must add before T-001.5** · **Add inside T-001.5** · **Defer to future stage** · **Reject / too much process**.

## Executive Verdict

**The core intent is right; the planned 15-artifact structure is over-built and, as drafted, re-triggers the exact failure mode this project already diagnosed and rejected.** The traceability chain (messy idea → source scan → selection → alternatives → assumptions → approach → evidence → failure handling → reuse → expansion → handoff → approval) and "input can be messy, execution must be professional" are genuinely worth codifying — once, lightly. But shipping 15 standing artifacts (source scan log, evidence matrix, framework selection matrix, assumption log, failure taxonomy, etc.) on top of a 20-row dummy-data project would roughly double the governance surface and contradict the project's own commitments: `RULES.md` §13 (lightweight loop for offline work), §8 (don't let process overshadow product), the governance review's "governance has outgrown the product" finding, and `plan-reconciliation.md` §4 which **explicitly rejected a docs-first prerequisite gate**. T-001.5 as 15 artifacts is a new docs-first gate.

**Recommendation: proceed — but collapse the 15 into ONE ~1–2 page playbook plus small fields added to existing files. Do not create 8–10 new living logs/matrices.** In reduced form it is high-value and low-cost; in maximalist form it is process theater.

## Missing Must-Add Rules

- **Source quality tiers + sufficiency/timebox** (answers Q5/Q12). `RULES.md` §6 says "verify or mark UNVERIFIED" but never defines *what counts as a good source* or *when research is enough*. Add a 3-line rule: prefer primary/official docs > reputable secondary > blog/forum > memory(never); cite the source next to load-bearing claims; **research is "enough" once every load-bearing platform/pricing/capability claim is sourced or marked UNVERIFIED — stop there** (anti-infinite-research). → **Add inside T-001.5.**
- **Git-state re-derivation as a *required* checklist step** (from the T-001 audit). The handoff git-state line has gone stale **3×**; the soft "be careful" prevention failed. Make "re-derive the PROJECT_STATE/HANDOFF git-state line from `git log -1` + `git status`" a checked item in `prevent-repeat-checklist.md`. → **Must add before T-001.5** (cheap, already-identified, recurring failure).

## Optional Improvements

- **Alternatives field in `decision-log.md`** — record "alternatives considered + why chosen," proportional to risk (one line for reversible/offline; a full ADR for architecture/integration). Covers #4/#6/#8 without a "matrix." → **Add inside T-001.5.**
- **Public-claims / portfolio rule** — consolidate the scattered honesty rules into one: demos/screenshots/metrics must be labeled *simulated*; never imply real DoorDash data, real integrations, or real business impact; never claim "no AI was used" or "AI built it." Mostly exists (§4/§7/§8); consolidating helps a portfolio reviewer. → **Add inside T-001.5.**
- **Merge the four `docs/prompts/*` templates into one** (the governance review already suggested this). → **Add inside T-001.5** (net-zero or net-negative file count).

## Rules That May Be Too Heavy

- **"Deep rationale" for every framework/principle/tool/approach (#7).** "Deep, always" is the bloat trigger. Reframe to **rationale proportional to risk/irreversibility**: one line for reversible/offline choices; a fuller ADR only for architecture, integrations, live sends, or publishing. → **Reject as written; Add inside T-001.5 in proportional form.**
- **"Evidence matrix" (#3) as a standing artifact.** The test suite already is the evidence. A separate matrix duplicates it and will drift. → **Reject.**
- **"Framework selection matrix" (#4).** A matrix implies many candidates per task; this project makes small, mostly-forced choices. → **Reject; use the decision-log alternatives field.**
- **"Failure taxonomy" (#10) as a new doc.** The prevent-repeat-checklist + journal already capture failures with root cause and prevention. A formal taxonomy is enterprise ceremony for a solo project. → **Reject the taxonomy doc; keep the existing checklist/journal.**
- **"Source scan log" / "assumption log" as standing files (#2,#5).** Per-task standing logs go stale (the audit just proved docs drift). → **Reject as files; Add inside T-001.5 as short *sections/fields* in the task brief.**

## Redundant or Overlapping Files

This is the strongest structural concern. The repo already carries a heavy governance surface for a 20-row CSV: `RULES`, `AGENTS`, `CLAUDE`, `CODEX`, `dual-model-workflow`, `prevent-repeat-checklist`, 4 prompt templates, `decision-log`, `implementation-journal`, `task-log`, `open-questions`, `project-narrative`, `product-brief`, `data-audit`, `plan-reconciliation`, 3 reviews, 3 audits, review packets, visuals. Adding ~15 more would roughly **double** it. Existing overlaps already flagged: `implementation-journal` vs `decision-log` (both capture "why"); the 4 prompt templates; HANDOFF block accretion (cleaned in the audit). **Recommendation: the playbook must be net-neutral or net-negative on file count — absorb scattered guidance, don't add alongside it.** → **Must add before T-001.5: a "no new standing logs unless a rule requires one" constraint on the playbook itself.**

## Conflicts With Flow-State Building

- The 15-artifact version conflicts directly with `RULES.md` §6/§13 flow-state intent: filling a source-scan log + evidence matrix + framework matrix + assumption log **per task** would gate every small offline change behind paperwork. The T-001 slice was ~3 small modules + tests; under the maximalist playbook it would have carried 5+ side artifacts. → the **lightweight tier must be the default**, and the full chain reserved for risky/integration/publish work.
- "Deep rationale always" conflicts with flow-state. Proportional rationale does not.

## Source-Backed Research Gaps

`RULES.md` §6 is the right spine but lacks: (a) **source tiers**, (b) a **sufficiency/stop rule** (anti-infinite-research), (c) **used-vs-rejected** capture, (d) **re-verify-on-staleness** for vendor/model/API/pricing claims (answers Q14). The Codex command surface was correctly verified against the installed plugin earlier — that's the standard to generalize, lightly. → **Add inside T-001.5** (a 4–6 line rule, not a research methodology).

## Framework and Tool Selection Gaps

No requirement today to record **alternatives considered and why rejected**, or **why now** for adding a tool/integration. Genuine gap, but solved by a decision-log field + the existing "add only when justified" roadmap principle — **not** a matrix. Tool/model/API/pricing/free-tier assumptions (Q14) should be recorded as **UNVERIFIED until checked against current vendor docs** (extends §6). → **Add inside T-001.5.**

## Planning Discipline Gaps

Largely covered (`claude-task-template`, slice plan, reconciliation). The one real gap is **use-case classification up front** (is this offline-deterministic vs integration vs publish?) — because that classification *selects the workflow tier*. Add it as the first field of the task template. → **Add inside T-001.5.** Beware: do not reintroduce a broad planning loop (the project's documented anti-pattern).

## Implementation Discipline Gaps

Well covered by the T-001 evidence (TDD-style tests, fix-logic-not-tests, deterministic/offline-first, idempotency, guardrails). No must-add. **Reusable-pattern capture (#13 / Q13)** is a minor gap → a one-line "note reusable patterns in the journal/decision-log" rule. → **Add inside T-001.5** (one line) or **Defer**.

## Validation and Acceptance Criteria Gaps

`RULES.md` §9 (definition of done) is per-task; there is no explicit **stage-close gate** (what closes T-001 vs a sub-task). The T-001 audit performed one ad-hoc. Formalize a short **stage-close checklist** (tests green, docs in sync with git, no secrets/clutter, follow-ups classified, human sign-off). → **Add inside T-001.5.** Do not add a separate "evidence matrix" — the stage-close checklist + tests suffice.

## Public Documentation / Portfolio Gaps (Q11)

Partly covered (§4/§7/§8), but no single rule for **demos/screenshots/metrics/visuals in a portfolio context**. A portfolio reviewer is the actual audience. Add one consolidated rule: every public artifact labels data *simulated*, shows no real credentials/integrations, and frames the dual-model method as *internal*. → **Add inside T-001.5.**

## Handoff and Continuity Gaps (Q8)

`HANDOFF.md` + standing procedures already serve Claude acct 1/2, CLI, Codex. Gaps: (a) the **git-state re-derivation requirement** (Must-add, above); (b) an explicit note that **ChatGPT web / external reviewers get the review-packet, not repo write access** (read-only context handoff). → git-state = **Must add before**; ChatGPT-handoff note = **Add inside T-001.5** (one line).

## New Information / Reuse / Blindspot Capture Gaps (#11 planned, Q13)

No current rule for "when you discover something that changes a prior decision, record it." Lightweight fix: **when new info invalidates a decision, log it in `decision-log.md` (or `open-questions.md`) and update the affected doc in the same task.** Reusable patterns → a one-line journal note. → **Add inside T-001.5** (2 lines). A standing "blindspot log" file is **Reject** (will go stale).

## Recommended Final T-001.5 Scope

**One document: `docs/enterprise-delivery-playbook.md` (~1–2 pages), plus small edits to 2 existing files.** The playbook:

1. States the **core traceability chain** (#15) as the spine, and **"professional but lightweight"** (#14) as the governing principle.
2. Makes the **use-case tier the first decision** and defines the tiers explicitly (extends `RULES.md` §13): *lightweight* = offline/deterministic/docs; *full* = integrations, live sends, auth/security, data writes, publishing.
3. Specifies the chain's fields as **sections in the (merged) task template**, proportional to tier: use-case classification → sources (tier + used/rejected + sufficiency stop) → selection + alternatives → assumptions → approach → validation (= tests) → failure/prevention (= existing checklist) → reuse note → expansion one-liner → handoff → human approval.
4. Adds the **source-tier + sufficiency/timebox** rule and the **vendor/model/API/pricing = UNVERIFIED-until-checked** rule.
5. Adds the **public-claims/portfolio** rule and the **new-info/reuse capture** one-liner.
6. Adds a short **stage-close checklist**.

Plus: **(a)** add the **git-state re-derivation step** to `prevent-repeat-checklist.md` (Must-add); **(b)** add an **"alternatives considered"** field to `decision-log.md`; **(c)** merge the four `docs/prompts/*` into one. Net file change: **+1 playbook, −3 prompt files = net −2.**

## What To Exclude From T-001.5

**Reject (process bloat):** standalone source-scan log, evidence matrix, framework selection matrix, standing assumption log, failure-taxonomy doc, standing blindspot log, "deep rationale always." **Defer to the stage that needs them:** integration security checklist (RLS/webhook-signing/least-privilege — already captured in `codex-initial-review` for later), operational cost/budget ledger rules, reliability/observability (retries/timeouts), the evaluation harness itself (that is T-002's candidate work, not a playbook rule), data-privacy/PII and accessibility (no real data / no UI yet).

## Final Recommendation

**Proceed to create T-001.5 — in the reduced single-playbook form above — after closing T-001's three open follow-ups** (the `out/` volatile-log policy, the `v1-slice-plan` test-list doc-sync, and the git-state checklist step), so the playbook isn't built on an unclosed stage. If the team insists on the full 15-artifact version, **do not proceed** — it would convert the project's strongest asset (disciplined, source-backed reasoning) into the liability it already warned against (governance outpacing product). The honest test for the playbook: it should make the *next* task faster and more traceable, not add paperwork to it.

Classification summary:
- **Must add before T-001.5:** git-state re-derivation checklist step; "no new standing logs" constraint on the playbook.
- **Add inside T-001.5:** the single playbook (traceability spine + tiers), source tiers + sufficiency rule, decision-log alternatives field, public-claims/portfolio rule, stage-close checklist, new-info/reuse one-liners, ChatGPT-read-only note, merge prompt templates.
- **Defer:** integration security/cost/reliability rules, eval harness, privacy/accessibility.
- **Reject:** the separate logs/matrices/taxonomy and "deep rationale always."
