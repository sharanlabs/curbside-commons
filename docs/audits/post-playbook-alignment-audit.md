## Recommended Next Stage

**Offline evaluation harness first (proposed T-002).** Build a small, fully-offline eval: golden blocker/risk labels for the 20 rows, a guardrail regression set (the planted + real-nudge cases + each category), and a metrics summary — no new integrations, no secrets, no cost.

Why this over Gemini: "evaluation before claims" is in `RULES.md` §3 and the playbook. The guardrail today passes *by construction* (stub generator); its real robustness is unproven without an eval set. An offline eval (a) strengthens the honesty story, (b) stays deterministic/offline/reversible (lowest risk), and (c) **de-risks the eventual Gemini swap** — once golden labels exist, replacing the stub with Gemini becomes *measurable* (you can prove the AI step doesn't regress the deterministic baseline). Gemini-first would introduce the first external API + secret + cost + non-determinism + freshness burden *before* any way to measure whether it helps — premature by the project's own ordering.

Note: this **deviates from the literal ordering** in `docs/plan-reconciliation.md` §6 (which puts Gemini next). The owner should ratify the eval-first reorder in `docs/decision-log.md` before T-002 begins (not done here — no decision is committed by an audit).
