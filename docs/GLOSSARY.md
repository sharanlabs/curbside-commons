# Glossary

The shared decoder ring for this project (per [`documentation-standard.md`](documentation-standard.md)). Every term of art used in the docs gets an entry: the **precise definition** first, then **in plain words**. Grows at first use of any new term — an undefined term in a doc is a defect.

*Sorted alphabetically. Last updated: 2026-07-07 (A1: MCP server — MCP (Model Context Protocol), MCP tool transcript added).*

| Term | Precise definition | In plain words |
| --- | --- | --- |
| **ACP (Agentic Commerce Protocol)** | OpenAI's published specification for merchant product feeds and checkout that AI agents (e.g., ChatGPT) consume to browse and transact; includes feed format and refresh-cadence requirements. | The standard format restaurants/stores use to tell AI assistants "here's my menu and prices" so the assistant can order. |
| **Advisory finding (classifier)** | An F1b classifier-derived relabeling *candidate*: built through the same C2 receipts guard (`makeFinding`, `claim.source: "classifier"`) but carried in a separate array that never merges into the base report's findings and never gates `report.ok`; severity `info`; labeled with its classifier's provenance and non-earned status. | A lead the AI flags for a human to check — "this fee might really be something else" — never a verdict, and it can't change the legal checker's answer by itself. |
| **Anti-theater floor (AM-7)** | A deterministic baseline whose measured performance an LLM component must BEAT on held-out gold, in an owner-gated live run, before its label ("classifier", "judge", "agent") is earned; ties or losses → the label DEFERS and the step stays a tool. The F1b keyword-rule baseline is this floor for the fee line-item classifier. | The dumb-but-honest benchmark the AI must visibly out-score on unseen examples before we're allowed to call it good — otherwise we don't. |
| **Asserted pass-through (unverified)** | The fee-audit verdict state `asserted-passthrough-unverified` (M2, 2026-07-04): a transaction fee above the 3% cap whose §20-563.3(c)(i)–(ii) pass-through exception rests solely on the platform's own `passthroughDocumented` flag. The statement cannot verify the fee equals the actual processor charge, so the audit surfaces the reliance as a non-gating warn — neither silently cleared nor called a violation. | The platform says an over-3% card fee is "just passing on the real card cost." The bill alone can't prove that either way, so the checker flags it for follow-up instead of trusting it or accusing anyone. |
| **Audit trail** | An append-only, structured record of every action, decision, and actor in the system, sufficient to reconstruct what happened and why. | A tamper-evident logbook: who did what, when, and on what evidence. |
| **Basic service fee** | NYC §20-563 (as amended by LL79/2025): the fee for the service that lets a food service establishment receive delivery/pick-up orders on all the platform's sites and be prominently searchable; capped at 5% of the purchase price per order, or on the calendar-month average (§20-563.3(b)). | The "being listed and findable on the app" fee — max 5%. |
| **Calibration (of a judge)** | Measuring an LLM judge's precision/recall/agreement against a labeled gold set before trusting its verdicts; the measurement is frozen and cited. | Testing the AI referee against answer-keyed examples first, and publishing its score. |
| **Canonical serializer** | A named function that is THE single sanctioned way to turn an engine report (or a tool's payload) into a string — e.g. `serializeReport`, `serializeFeeReport`, or (new at A0) `serializeClassifiedFeeReport`/`serializeRuleLookup` (`lib/tools/serializers.ts`). Because it is the only allowed stringifier, calling it twice on the same data always gives byte-identical output — which is what makes a differential test ("does the registry's answer match a direct engine call?") a strict string-equality check. | The one exact way an answer gets printed as text, always the same way — so two different code paths that are supposed to agree can be checked byte-for-byte, not just "looks about right." |
| **Category lock** | §20-563.3(d) opening clause: a third-party delivery service may charge a food service establishment only the four permitted fee categories (delivery · basic service · transaction · enhanced service); any other fee is unlawful. | Only four fee types are legal — anything else on the statement is illegal, no matter what it's called. |
| **Client-side schema composition** | The UCP requirement that a validator assemble ("compose") a payload's full schema locally from cross-referencing files (relative `$ref` resolved against each file's absolute `$id`), rather than fetching one monolithic schema. W2's conformance leg registers every pinned UCP schema by `$id` so ajv resolves the whole graph offline. | The official rulebook comes as many small linked files; the checker gathers all the linked pieces itself before it can judge a document. |
| **Command-level dry run** | The A4 veracity test (`evals/n8n/n8n-workflow.test.ts`): the committed n8n workflow's OWN command strings are extracted from the JSON and executed verbatim from the repo root, and the final payload artifact must byte-equal a direct registry→builder build. Proves the automation lane's substance without the n8n runtime; running the same workflow UNDER n8n is the owner-gated O-A4/L-3 leg. | The robot's to-do list is run line by line exactly as written, and the result must match doing it by hand — so the list is proven real even before the robot app itself is switched on. |
| **Conformance** | The degree to which an implementation satisfies the normative requirements of a specification (MUST/SHOULD clauses). | Whether your feed actually follows the rules of the standard it claims to follow. |
| **Conformance vs truth** | Conformance = the feed matches the SPEC (fields, types, schema). Truth = the feed matches REALITY (the SOR). A feed can be 100% spec-valid and still wrong. | Passing the form-check isn't the same as telling the truth on the form. We check both, and the second is the point. |
| **Conformance-foil (demo beat)** | The D1 demo's closing beat: the SAME UCP catalog-response document is run through the conformance leg (it PASSES — spec-valid) and then the truth leg against the SOR (it is caught misstating a price). The beat's line: *passes the official schema check; still lies.* Both verdicts are computed from the real check results, never narrated. | The demo's punchline: a menu answer can follow the official rulebook perfectly and still quote a wrong price — and the checker shows exactly that. |
| **Cross-family (models)** | Using models from different vendors/architectures for maker vs judge roles (e.g., Gemini drafts, Groq-hosted GPT-OSS judges) to mitigate self-preference bias. | The player and the referee come from different teams, so the referee isn't biased toward its own side. |
| **Declared vs true category** | On a fee statement, the *declared* category is what the platform CALLS a fee line (which may be a legal category or an invented label); the *true* category is what the charge actually is. The F1a deterministic spine audits categories **as declared**; recovering the true category from a mislabeled line is the F1b LLM line-item classifier's job. | What the bill says a fee is, versus what it really is — the deterministic checker trusts the label; the AI classifier (next slice) sees through it. |
| **Deferred-to-classifier** | A planted violation the deterministic spine cannot catch from a single monthly statement (e.g. a bundle hidden under a legal within-cap label, or pure cross-month relabeling), recorded honestly in the answer key as needing the F1b classifier or multi-month/notice data — never faked as deterministically caught. | A cheat our exact-logic checker genuinely can't see yet, marked as such instead of pretended-caught. |
| **Deterministic verification** | Comparison logic with a single, repeatable, rule-defined outcome for given inputs — no ML, no sampling, no variance. | Exact checking, like arithmetic: same inputs, same answer, every time. Can't hallucinate. |
| **Differential oracle** | A second, independent implementation run over the same inputs whose verdicts must AGREE with ours — a check on our own checker. W2's is the official cargo-only `ucp-schema` Rust validator, run in CI when present and SKIPPED-LOUD (agreement UNMEASURED) when absent — never a runtime dependency. | A second referee from a different toolmaker; if the two ever disagree, we have a bug to find. |
| **Delivery payload builder** | An A3 pure function (`lib/delivery/slack.ts` / `email.ts`) from a registry tool's canonical report string to a transmittable payload (Slack Block Kit JSON / RFC 5322 MIME) — deterministic, byte-frozen by goldens, SIMULATED-bannered at build time, with NO transport/credentials/addresses beyond `.example` placeholders. Sending = the owner-gated L-2 transient demo under `docs/plan-a3-delivery-safety.md`. | Writes the Slack post or email a deployment WOULD send — but has no send button, no address book, and stamps SIMULATED on every message it writes. |
| **Drift** | Any divergence between a serving copy (feed, listing, agent-visible catalog response, fee statement line) and the SOR — price, availability, existence, identity, staleness, encoding, version skew. | The copy no longer matching the original — wrong price, item that's actually sold out, a fee that isn't what was agreed. |
| **Drift injector** | The W1 tool that takes a truthful serving copy and deterministically plants documented drift into it, one taxonomy class per row, recording every mutation in a ground-truth manifest. | The lie-planter: it breaks the copy of the menu in each documented way, and writes down every break so tests know exactly what must be caught. |
| **Drift-lock** | An eval that binds the codified TS rule predicates to the JSON rule-table twin by set-equality both directions: every twin rule id is either implemented (with cap_pct/base/drift_classes matching 1:1) or registered non-statement-checkable with a reason; an unregistered-and-unimplemented id fails, and a TS rule absent from the twin fails. | A test that makes it impossible for the code and the legal rulebook to quietly drift apart — change one without the other and CI stops you. |
| **Non-statement-checkable rule** | A codified rule the fee-statement schema cannot verify because it needs data an itemized statement does not carry (e.g. fee-change notice dates for g-1-iv, or a service/searchability obligation) — registered with a written reason rather than faked. | A real legal rule our checker honestly can't test from a bill alone, listed and explained instead of pretended-checked. |
| **Enhanced service fee** | NYC §20-563 (LL79/2025): a fee for services beyond the basic service, chargeable only if the platform also offers the basic-service tier; capped at 20% of the purchase price per order, or on the calendar-month average (§20-563.3(d)). | Optional extras (e.g., marketing) — max 20%, and only allowed if a plain basic plan exists too. |
| **Entity resolution (ID-mismatch)** | Recovering which truth row a serving-copy row refers to when identifiers don't line up (e.g. a legacy POS id vs the catalog id) — in W1, a deterministic exact-name heuristic; a required C3 drift class. | Figuring out "this is actually the same burger" when the copy calls it by a different id. |
| **Evals / gold set** | A curated, labeled test suite with known-correct answers used to measure system quality; the gold set is the answer key, evals are the exams run against it. | Answer-keyed practice tests we grade the system on, so quality is measured, not vibes. |
| **Evidence-cited (finding)** | A reported discrepancy that carries machine-checkable pointers to both sources (e.g., row IDs, timestamps, rule/policy versions) supporting the claim. | Every accusation comes with receipts attached. |
| **Eval-lock** | Freezing a live run's full record (per-item + metrics + verdict) into a committed snapshot, plus an offline regression test that recomputes the headline numbers from the per-item records and asserts the verdict state — never re-running live (R-DHON-4 pattern). The F1b run #2 lock is `evals/gold/fee-classifier-calibration.lock.test.ts`. | The score sheet is photocopied into the repo and a test keeps re-adding it up — so nobody can quietly edit the grade, and "just run it again" can't replace what actually happened. |
| **Fee cap** | A statutory ceiling on the commission/fees a delivery platform may charge merchants in a jurisdiction (e.g., NYC Local Law 79/2025's tiered caps). | A legal maximum on what the app is allowed to take from a restaurant's sales. |
| **Freeze-integrity (of a corpus)** | An eval asserting the committed fixture files are byte-identical to the seeded generator's output, so the corpus cannot be hand-edited without CI failing. | Proof the demo data wasn't quietly tampered with: rebuild it from the recipe and it must match to the byte. |
| **Golden report / golden fixture** | A frozen expected-output file the test suite byte-compares against the live output (possible because the verifier's output ordering is deterministic). | The answer sheet: the exact report the verifier must produce, saved, and compared character-for-character. |
| **HITL (human-in-the-loop)** | A control pattern where the system recommends and a human approves before any consequential action executes. | The tool suggests; a person decides. Nothing fires on its own. |
| **Hybrid data** | A dataset combining real open-source records with a synthetic overlay, each provenance-labeled. | Real public data as the base, made-up (and clearly labeled) details layered on top. |
| **LLM judge** | A language model used to evaluate outputs or resolve fuzzy equivalences, always calibration-measured and never the sole gate on consequential actions. | An AI referee for judgment calls — used narrowly, and only after we've scored how reliable it is. |
| **Machine-JSON leg** | The verifier CLI's output contract: the full `VerifierReport` serialized as canonical `JSON.stringify(report, null, 2)` on stdout — CI-usable (a job can parse it and gate on the exit code), always carrying the C10 header surface (specVersion · matchingMode · simulated). `--json` is an explicit trailing alias for this default serialization; exit codes 0/1/2 are frozen. | The result as structured data a robot can read and act on — the same report the web page shows, in a form a build pipeline can fail on. |
| **Maker ≠ judge** | The independence principle that the party producing an artifact must not be the party verifying it — applied at ecosystem scale (sync vendors shouldn't audit their own syncing) and at model scale (cross-family judging). | You don't grade your own homework. |
| **MCP (Model Context Protocol)** | An open, Linux Foundation-governed standard (official TypeScript SDK `@modelcontextprotocol/sdk`, MIT) letting an AI client discover and call a fixed set of named, schema-typed tools over a transport — here, `stdio` only (never a network socket). A1's server (`lib/mcp/server.ts`) exposes the A0 tool registry's six tools over MCP via the SDK's low-level `Server` class, advertising each tool's COMMITTED JSON Schema verbatim (not hand-converted to a different shape) and mapping every call through the registry's own `callTool` — never a tool's `run()` directly. | The standard plug shape that lets an AI assistant (or any MCP-speaking tool) press this project's six checker buttons itself, over a private local pipe, using the exact same rulebook the checker already validates against. |
| **MCP tool transcript** | A committed, byte-frozen JSON record of a REAL scripted MCP client session (initialize → tools/list → one call per tool → invalid calls) against a REAL spawned server process — never a mock. Normalized for determinism: no JSON-RPC request ids (a connection-timing artifact, not part of the tool contract), no absolute paths, no timestamps/PIDs, and every advertised input schema canonicalized (recursive key-sort) to strip the MCP SDK's own harmless key-reordering. A freeze test asserts regenerating it reproduces the committed bytes exactly. | A frozen, provable "here's exactly what happened" recording of a real conversation with the plug-in server — proof it behaves as documented, not just a claim. |
| **Member × case matrix** | The committed per-case result table of an A2 trajectory replay (`evals/crew/gold/member-case-matrix.golden.json`): for every committed case — safety invariants passed?, expected class matched?, terminal state. The §6 floors are judged on this matrix PER MEMBER, never as one blended average, so one member's failure cannot hide inside a passing crew. | Each teammate gets its own report card, question by question — no grading on the group curve. |
| **Monthly-average alternative** | §20-563.3(a)/(b)/(d): the cap may be satisfied on the calendar-month average — Σ(fees in month) ≤ cap% × Σ(purchase prices in month) — instead of per order. The transaction fee (c) has no averaging alternative. | Passing the cap on the month's average instead of every single order — allowed for three of the four fee types, never for the 3% processing fee. |
| **One-page report (S-9)** | The verifier's output as a single self-contained document — evidence-cited, spec-version pinned, simulated-labeled — rendered two ways over the SAME data: a machine-JSON leg (CLI) and a web view (`/report`). "One page" means one cohesive, printable document per report, not a hard one-physical-page cap; every finding's four receipts stay visible (C2 forbids hiding them), and the print stylesheet keeps a finding from splitting across a page break. | The result written up as one readable (and printable) page — plain sentence per catch, receipts beside it — nothing hidden to make it fit. |
| **Orchestration harness passed** | The ONLY label an offline trajectory replay can earn (plan §6): the crew's containment machinery (tool gating at the call site, demo/advisory refusal, human-gate forcing, finding-reference validation) held on every committed case, with model turns replayed from a committed fixture. Explicitly NOT the "agent" label — that requires the owner-gated live run (L-1) clearing the same per-member floors on a held-out split. | "The rails and brakes all work." It says nothing yet about the driver — the AI wasn't really driving. |
| **Over-cap refund safe harbor** | §20-563.3(e): an over-cap charge under subdivisions a/b/d is not a violation if the excess is refunded within 30 calendar days of the final day of the month of the charge; subdivision c (transaction fee) is excluded. | A 30-day grace window to refund an overcharge — except on the 3% processing fee, which gets no grace at all. |
| **Pre-build crux gate (G8)** | The hard gate (Codex amendment 1) that blocks UC-2 implementation until the two-part crux is resolved: (i) does a copy layer persist behind live catalog reads in marketplaces? (ii) who has authority/pain/budget to require independent truth evidence? | Before building, we must prove the problem will still exist (copies keep drifting) and that someone would actually run the tool. |
| **Capture-then-replay (L-1)** | The live-run mechanic that keeps shipped code untouched: each live model decision is fetched ONCE and recorded, then the authoritative trajectory record is produced by re-running the UNCHANGED orchestrator (`runCase`) with those recorded decisions — the same code path as the offline replay. The committed raw turns make the whole result re-derivable offline forever (`l1-live-lock`). | The AI's answers are written on cards first; then the exact same exam machine that ran the practice test runs them — so the "live" grade can be re-checked any time without calling the AI again. |
| **Deterministic param mapper (L-1)** | The §6-addendum rule that a live model never authors tool parameters: its decision surface is a closed vocabulary (pick a tool / reject; approve / escalate), and parameters derive mechanically from the tool name + the case's committed artifact path. The mapper never reads the case's expected calls, so the orchestrator's digest containment independently verifies mapper ≡ contract. | The AI may only point at which button to press — the machine fills in the button's settings itself, from a fixed rulebook the AI can't touch. |
| **Pre-registration (calibration)** | Pinning the pass/fail floors, decision rule, and harness semantics in a committed doc BEFORE a live run executes; floors may only ever tighten pre-run, never move after results are seen; a missed floor ⇒ the label DEFERS (no same-split re-run under the 2026-07-04 arming amendment). | Writing the grading rules down before the exam, in ink — so the bar can't move once the score is known. |
| **Precision / recall** | Precision: of the items flagged, the fraction truly wrong. Recall: of the items truly wrong, the fraction flagged. | Precision = how often the alarm is right. Recall = how many real problems the alarm catches. |
| **Print stylesheet** | A CSS `@media print` block that restyles a page for paper/PDF output — here it hides the nav, footer, and surface toggle, forces the honesty labels and severity colors to print (`print-color-adjust: exact`), and marks each finding `break-inside: avoid` so a catch and its receipts never split across a page break. | The extra styling that makes the on-screen report print to a clean page, with the "simulated" label always visible. |
| **Purchase price base** | "Purchase price of each online order" — the statutory base every cap percentage is measured against (§20-563.3(a)–(d)). Its inclusions/exclusions (tax, tip, pre- vs post-discount) are UNVERIFIED as of 2026-07-03 (source-memo U1) — base-derived verdicts are provisional until resolved. | The order amount the percentages apply to — with an honest asterisk: exactly what counts inside it is still an open question. |
| **Reconciliation** | The operational process of comparing two independent records of the same activity (e.g., POS sales vs platform payout statement) and explaining every difference. | Lining up two versions of the books and accounting for every mismatch, penny by penny. |
| **Recorded model turn** | A committed, typed decision (`evals/crew/cases/recorded-turns.json`) replayed in place of a live model call during offline trajectory evaluation; an unplanned turn throws loudly rather than defaulting. Steered/hostile recorded turns deliberately simulate a compromised model so what gets measured is the ORCHESTRATOR'S containment. | The "AI's answer" comes from a checked-in answer sheet — including some deliberately WRONG answers, to prove the guardrails catch a misbehaving AI. |
| **Refund-window verdict state** | The disposition F1a assigns an over-cap on a/b/d, encoded (not prose): `violation` (window closed, no covering refund), `conditional-pending-refund-window` (window still open — not yet a violation), or `cured-by-refund` (refunded in time — not a violation). The transaction fee (c) has no such state — it is a violation immediately (no §20-563.3(e) safe harbor). | Whether an overcharge is a settled violation, still inside the legal refund-grace clock, or already refunded and cleared — a real state, not a fixed yes/no. |
| **SOR (system-of-record)** | The authoritative data source for a fact domain — the merchant's POS/catalog for menu truth, the payout ledger for money truth — against which every served copy or statement line is verified. | The one place that holds the *real* answer — the till, not the app's copy. |
| **Spec-faithful demonstration actor** | The D1 demo's scripted stand-in agent — labeled "spec-faithful demonstration actor — simulated." It consumes ONLY the published serving copy (its transitive imports are machine-proven to exclude the SOR reference resolver and every SOR fixture — SOR-blindness), applies one fixed scripted intent, and selects the target item at the copy's face value. Deterministic, $0, no LLM; the demo ends at selection (slice-C cut, no checkout). | A rule-following pretend shopping agent that only ever sees the published menu (never the restaurant's till), decides by a fixed rule, and believes whatever the menu says — used to show the drift it can't detect. |
| **Synthetic data** | Deliberately constructed data (including adversarial edge cases), clearly labeled as simulated, used for demos and eval coverage. | Made-up test data — labeled as such — built to include the tricky cases real data rarely shows on demand. |
| **Take-rate (effective)** | The total share of merchant revenue a platform captures once all fees (commission, processing, marketing, promos, refund costs) are included — vs the headline commission. | What the platform *really* takes, not the advertised percentage. |
| **Tool envelope** | The typed, frozen result every A0 registry tool returns (`ToolResult`, `lib/tools/types.ts`): which tool ran, whether it passed (`ok`, defined as `exitCode === 0` — enforced at construction), its exit code, the honesty flags a consumer must never miss (`demoOnly` on `run_demo`; `advisory`/`earnsLabel` on `classify_and_audit`), and `canonical` — the tool's own named-serializer string output. | The same-shaped little receipt every tool hands back — which tool ran, did it pass, and (when it matters) a flashing "this is a demo" or "this is just a suggestion" flag. |
| **Tool registry** | The A0 seam (`lib/tools/registry.ts`): six typed JSON-in/JSON-out tools (`check_feed`, `check_conformance`, `audit_statement`, `classify_and_audit`, `get_rule`, `run_demo`), each wrapping an UNCHANGED engine entry point with a committed input/output JSON Schema. `callTool(name, params)` validates `params` against the tool's schema via ajv BEFORE running it — invalid input fails loud with a typed `ToolInputError`; an unregistered name fails loud with `ToolNotFoundError`. This is the ONE call surface every later surface (MCP server, agent crew, delivery builders, n8n lane) is meant to consume, so they can never bypass the engine's own validation and error discipline. | The one panel of six clearly-labeled buttons every future robot (an AI agent, an automation workflow) presses instead of reaching into the checker's engine room directly — press the wrong button, or press it with a wrong-shaped input, and it refuses loudly instead of guessing. |
| **ToS-clean access** | Reading data only through channels the terms of service permit: public documents, one's own accounts/sandboxes, and merchant-permissioned APIs — never scraping walled surfaces. | Only looking where we're allowed to look. |
| **Trajectory case** | One committed JSON document (schema in `docs/plan-a2-trajectory-floors.md`) specifying a crew run's contract and expectations: input artifact, allowed/forbidden tools, the exact expected executed tool-call sequence (with sha256 argument digests), the expected engine-report hash, recommendation class, finding references, and gate state. The unit the §6 floors are measured on; "the injection does not steer tool choice" is DEFINED as the actual sequence still matching this document. | One exam question for the helper team, written down and locked before the exam — what it's given, what it may touch, and exactly what a right answer looks like. |
| **Transaction fee** | NYC §20-563.3(c): the card-processing-type fee; capped at 3% of the purchase price per order — no monthly averaging, no refund safe harbor; above 3% only as an exact, documented pass-through of the actual processor charge, with proof available to the department and the establishment on request. | The card-processing fee — a hard 3%, unless they can prove every cent above it is a pure pass-through of real cost. |
| **Tune/test split** | A disjoint partition of a gold set: iteration/threshold-tuning happens ONLY on the *tune* half; every reported claim is scored ONLY on the held-out *test* half. F1b's fee-lines gold set pins 21/21 with per-class stratification and a composition test enforcing the discipline. | Practice questions vs the real exam — you may study the first pile all you want, but your grade only ever comes from the second. |
| **UCP (Universal Commerce Protocol)** | Google-led open agentic-commerce spec (unveiled 2026-01-11 at NRF; Apache-2.0; spec 2026-04-08 live at ucp.dev/GitHub; RFC-2119 normative language). Food vertical co-developed by DoorDash, Square, Toast, Uber Eats — food schemas pending. Catalog capability is a LIVE-QUERY interface (session-scoped), not a static feed. | The Google-side standard AI assistants use to shop and order. Its menu-lookup part asks the store live, rather than reading a saved copy — which matters for what can drift. |
